# <a name="implementing-least-privilege-administrative-models"></a>Implementieren von Verwaltungsmodellen der geringste Rechte

>Gilt für: Windows Server 2016, Windows Server 2012 R2, Windows Server 2012

Der folgende Auszug ist von [der Administrator Accounts Security Planning Guide](https://technet.microsoft.com/library/cc162797.aspx), veröffentlicht am 1. April 1999:

> "Schulungen und Dokumentation für die meisten sicherheitsbezogenen erläutern die Implementierung des ein Prinzip der geringsten Rechte, doch Organisationen kaum darauf folgen. Das Prinzip ist einfach, und die Auswirkungen der Anwendung ordnungsgemäß erheblich erhöht die Sicherheit und reduziert das Risiko. Das Prinzip besagt, dass alle Benutzer mit einem Benutzerkonto anmelden müssen, die absoluten minimalen erforderlichen Berechtigungen zum Abschließen der aktuellen Aufgabe und nicht mehr verfügt. Auf diese Weise bietet Schutz vor böswilligem Code und andere Angriffe. Dieses Prinzip gilt für Computer und dem Benutzer diese Computer.   
> "Ein Grund, die dieses Prinzips ist darin, dass Sie einige internen Untersuchungen durchführen. Sie müssen z. B. bestimmen, welche Zugriffsrechte, die einen Computer oder Benutzer wirklich benötigt und sie implementieren. Für viele Organisationen mag diese Aufgabe zunächst wie eine große Menge an Arbeit; Es ist jedoch ein wichtiger Schritt, um Ihre Netzwerkumgebung erfolgreich zu schützen.
> "Sie sollten alle Domänenbenutzer Administrator ihre Domänenberechtigungen nach dem Prinzip der geringsten Rechte gewähren. Wenn ein Administrator mit einem privilegierten Konto anmeldet, führt versehentlich ein Virusprogramm hat der Virus beispielsweise administrativen Zugriff auf den lokalen Computer und auf die gesamte Domäne. Wenn der Administrator stattdessen mit einem nicht privilegierten Konto für die (nicht-Administratoren) angemeldet haben, wäre der Virus-Bereich der Beschädigung nur dem lokalen Computer, da er als Benutzer eines lokalen Computer ausgeführt wird.
> "In einem anderen Beispiel Konten, die Sie auf Domänenebene Administratorrechte erteilen, müssen nicht Rechte in einer anderen Gesamtstruktur erweitert haben, auch wenn zwischen den Gesamtstrukturen eine Vertrauensstellung besteht. Dieses Vorgehen hilft Schäden zu verhindern, wenn ein Angreifer eine verwaltete Gesamtstruktur gefährden. Organisationen sollten regelmäßig mit ihrem Netzwerk zum Schutz gegen nicht autorisierte Ausweitung von Berechtigungen überwachen."  

Der folgende Auszug ist von der [Microsoft Windows Security Resource Kit](https://www.microsoftpressstore.com/store/microsoft-windows-security-resource-kit-9780735621749), erste veröffentlichte in 2005:  

> "Immer der Sicherheit in Bezug auf die minimal erforderlichen Privilegien zum Ausführen der Aufgabe erforderlich gewähren vorstellen. Wenn eine Anwendung, die zu viele Berechtigungen verfügt, gefährdet sein sollte, der Angreifer möglicherweise erweitern den Angriff über Was wäre wenn die Anwendung unter den geringsten Berechtigungen möglich gewesen wäre. Überprüfen Sie z. B. die Konsequenzen, ein Netzwerkadministrator unwissentlich Öffnen der Anlage einer e-Mail, die einen Virus startet. Wenn der Administrator mit der Domäne Administratorkonto angemeldet ist, wird der Virus über Administratorrechte auf allen Computern in der Domäne verfügen und uneingeschränkten daher Zugriff auf nahezu alle Daten im Netzwerk. Wenn der Administrator angemeldet ist mit einem lokalen Administratorkonto an, der Virus wird auf dem lokalen Computer über Administratorrechte verfügen und daher können zum Zugriff auf alle Daten auf dem Computer, und installieren schädlichen Software wie Software von Schlüssel-Stroke-Protokollierung auf der Computer. Wenn der Administrator mit einem normalen Benutzerkonto angemeldet ist, wird der Virus haben nur Zugriff auf Daten von der Administrator und ist nicht möglich, schädliche Software zu installieren. Mithilfe der mindestens erforderlichen Berechtigungen zum Lesen von e-Mail-Adresse, in diesem Beispiel wird der mögliche Bereich des Angriffs erheblich reduziert."  

## <a name="the-privilege-problem"></a>Das Berechtigungsproblem

In der vorherigen Auszüge beschriebenen Prinzipien wurden nicht geändert, aber bei der Bewertung von Active Directory-Installationen, suchen wir ausnahmslos eine übermäßige Anzahl von Konten, die Rechte und Berechtigungen, die weit über die erforderlichen tägliche Durchführung gewährt wurden verwendet werden. Die Größe der Umgebung wirkt sich auf die reine Anzahl an übermäßig berechtigte Konten, jedoch nicht auf die Verzeichnisse Proportionmidsized möglicherweise Dutzende von Konten in den meisten sehr privilegierten Gruppen, während große Installationen möglicherweise Hunderte oder sogar Tausende. Mit wenigen Ausnahmen ist unabhängig von der Komplexität eines Angreifers Fähigkeiten und Arsenal führen Sie Angreifer in der Regel auf den Weg des geringsten Widerstands. Sie erhöhen die Komplexität der Ansatz und Tools, nur wenn einfachere Mechanismen ausgefallen sind oder von Verteidiger ferngehalten.  

Leider ist der Weg des geringsten Widerstands in vielen Umgebungen belegt, dass die übermäßige Nutzung von Konten mit Berechtigungen für breit angelegte und tief. Umfassende Berechtigungen werden Rechte und Berechtigungen, mit denen ein Konto, um bestimmte Aktivitäten z. B. über eine große Querschnitt von der Umgebung ausführen, können Helpdesk-Mitarbeiter Berechtigungen erteilt werden, die sie für das Zurücksetzen der Kennwörter für viele Benutzerkonten zu ermöglichen.  

Umfassende Berechtigungen sind umfangreiche Berechtigungen, die einem schmalen Segment der Auffüllung, z.B. erhalten einen Techniker über Administratorrechte auf einem Server, damit diese Reparaturen können angewendet werden. Weder umfassende Berechtigungen als auch detaillierte Berechtigungen unbedingt gefährlich ist, aber wenn viele Konten in der Domäne dauerhaft breit angelegte und tief Berechtigung gewährt werden, wenn nur eines der Konten gefährdet ist, sie können schnell verwendet werden, die Umgebung neu zu konfigurieren der Zweck des Angreifers oder sogar große Teile der Infrastruktur zu zerstören.  

Pass-the-Hash-Angriffen, die eine Art von Angriffe mit gestohlenen Anmeldeinformationen sind, sind allgegenwärtig, da die Tools zu deren Ausführung kostenlos verfügbar und einfach zu bedienende ist und viele Umgebungen für Angriffe anfällig sind. Pass-the-Hash-Angriffen, sind jedoch nicht das eigentliche Problem. Der Knackpunkt des Problems bietet zwei Vorteile:  

1. Es ist in der Regel einfach für einen Angreifer erhalten umfassende Berechtigungen auf einem einzelnen Computer und dann diese Berechtigung Allgemein auf andere Computer zu übertragen.  
2. Es gibt in der Regel zu viele dauerhafte Konten mit hohen Berechtigungen für die IT-Landschaft.

Auch wenn die Pass-the-Hash-Angriffen behoben wurden, würden Angreifer einfach verschiedene Taktiken, nicht auf eine andere Strategie verwenden. Anstatt Pflanzen Malware, die den Diebstahl von Anmeldeinformationen Tools enthält, können sie werksstandort Malware, die Tastatureingaben protokolliert, oder nutzen eine beliebige Anzahl von anderen Ansätze, um Anmeldeinformationen zu erfassen, die in der gesamten Umgebung leistungsstark sind. Unabhängig von der Taktiken, die Ziele unverändert: Konten mit Berechtigungen für breit angelegte und tief.  

Eine übermäßige Rechte gewähren nicht nur in Active Directory in gefährdeten Umgebungen gefunden. Wenn eine Organisation sich mehr Berechtigungen als nötig erteilt entwickelt hat, ist es in der Regel in der gesamten Infrastruktur wie in den folgenden Abschnitten erläutert gefunden.  

## <a name="in-active-directory"></a>In Active Directory

In Active Directory ist es üblich, feststellen, dass die EA-Kunden, DA und BA-Gruppen eine übermäßige Anzahl von Konten enthalten. In den meisten Fällen enthält der Organisation EA-Gruppe die wenigsten Mitglieder, DA Gruppen in der Regel einen Multiplikator, der die Anzahl von Benutzern in der Gruppe "EA" enthalten und Administratorgruppen in der Regel enthält mehr Elemente als die Auffüllungen von den anderen Gruppen, die kombiniert. Dies ist häufig aufgrund von dem glauben, die Administratoren aus irgendeinem Grund sind "weniger privilegierten" als DAs oder EAs. Während die Rechte und Berechtigungen für jede dieser Gruppen unterscheiden, sollten sie effektiv gleichermaßen Gruppen betrachtet werden, denn ein Mitglied einer Trick Mitglied der beiden anderen vornehmen kann.  

## <a name="on-member-servers"></a>Auf Mitgliedsservern

Wenn wir die Mitgliedschaft der Gruppe der lokalen Administratoren auf Mitgliedsservern in vielen Umgebungen abrufen, finden wir Mitgliedschaft, die von der eine Reihe von Konten für lokale und Domänenprinzipale, bis hin zu Dutzenden von geschachtelten Gruppen, wenn die Kategorie erweitert ist, zeigen Sie Hunderte, sogar Tausende von Konten mit lokalen Administratorrechten auf den Servern. In vielen Fällen sind Domänengruppen mit großen Gruppenmitgliedschaften Mitgliedsserver lokale Administratoren Gruppen ohne Berücksichtigung der Tatsache geschachtelt, dass jeder Benutzer, die die Mitgliedschaften dieser Gruppen in der Domäne ändern kann, administrative Kontrolle über alle Systeme erlangen können auf dem die Gruppe in einer lokalen Administratorgruppe geschachtelte wurde.  

## <a name="on-workstations"></a>Auf Arbeitsstationen

Obwohl Arbeitsstationen in der Regel deutlich weniger Elemente in ihre lokalen Administratorengruppe sind als Mitgliedsserver ist, klicken Sie in vielen Umgebungen tun werden Benutzer die Mitgliedschaft in der lokalen Gruppe Administratoren auf ihren eigenen Computern gewährt. In diesem Fall auch dann, wenn UAC aktiviert ist, stellen Benutzer eine erhöhte Risikostufe für die Integrität von ihren Arbeitsstationen.  

> [!IMPORTANT]  
> Sollten Sie sorgfältig überlegen, ob die Benutzer benötigen Administratorrechte auf ihren Arbeitsstationen aus, und wenn dies der Fall ist, möglicherweise ein besserer Ansatz, ein separates lokales Konto auf dem Computer zu erstellen, die Mitglied der Gruppe "Administratoren" ist. Wenn Benutzer die Erhöhung der Rechte benötigen, können sie die Anmeldeinformationen des lokalen Kontos benutzeraufforderung nach erhöhten Rechten vorhanden, aber da das Konto lokal ist, kann nicht es auf andere Computer gefährden oder Zugriff auf Domänenressourcen verwendet werden. Wie bei der alle lokalen Konten, sollten jedoch die Anmeldeinformationen für das Konto mit lokalen Berechtigungen sein eindeutig sein. Wenn Sie ein lokales Konto mit den gleichen Anmeldeinformationen auf mehreren Arbeitsstationen erstellen, setzen Sie die Computer, die Pass-the-Hash-Angriffen.  

## <a name="in-applications"></a>In Anwendungen

Bei Angriffen, die in denen das Ziel geistiges Eigentum der Organisation ist, können Konten, die umfangreiche Berechtigungen innerhalb von Anwendungen gewährt wurden vorgesehen um Exfiltration von Daten zu ermöglichen. Auch wenn die Konten, die haben Zugriff auf sensible Daten, ohne erweiterten Berechtigungen einrichten, in der Domäne oder das Betriebssystem erteilt worden sind, bietet Konten, die die Konfiguration einer Anwendung oder den Zugriff auf die Informationen der Anwendung ändern können Risiko der vorhanden.  

## <a name="in-data-repositories"></a>In den Datenrepositorys

Wie bei anderen Zielen der Fall ist, können Angreifer Zugriff auf geistiges Eigentum in Form von Dokumenten und anderen Dateien suchen der Konten anzugreifen, Steuern des Zugriffs auf die Datei gespeichert werden, Konten, haben direkten Zugriff auf die Dateien, oder auch Gruppen oder Rollen, die sind, Zugriff auf die Dateien. Beispielsweise wenn Sie ein Dateiserver zum Speichern von Dokumenten für Vertrag verwendet wird, und der Zugriff wird gewährt, auf die Dokumente durch die Verwendung von Active Directory-Gruppen, kann ein Angreifer, der die Mitgliedschaft der Gruppe ändern, kann kompromittierte Konten zur Gruppe hinzufügen und Zugriff auf den Vertrag Dokumente. In Fällen, in denen Zugriff auf Dokumente von Anwendungen wie SharePoint bereitgestellt wird, können Angreifer die Anwendungen abzielen, wie oben beschrieben.  

## <a name="reducing-privilege"></a>Reduzieren von Berechtigungen

Je größer und komplizierter einer Umgebung, desto schwieriger ist das Verwalten und schützen. In kleinen Unternehmen überprüfen und reduzieren die Berechtigung können eine relativ einfache Angelegenheit sein, jede zusätzliche Server, Arbeitsstationen, Benutzerkonto und Anwendung in einer Organisation verwendet fügt jedoch ein anderes Objekt an, das gesichert werden müssen. Da es schwierig oder sogar unmöglich, ordnungsgemäß sein kann, Sichern Sie alle Aspekte einer Organisation IT-Infrastruktur, Sie sollten zuerst auf die konzentrieren Konten, deren Berechtigungen erstellen, das größte Risiko, das in der Regel sind der integrierten privilegierten Konten und Gruppen in Active Directory und privilegierten lokalen Konten auf Arbeitsstationen und Mitgliedsservern.  

### <a name="securing-local-administrator-accounts-on-workstations-and-member-servers"></a>Schützen lokaler Administratorkonten auf Arbeitsstationen und Mitgliedsservern

Obwohl dieses Dokument konzentriert sich auf Active Directory sichern, wie zuvor bereits erwähnt, beginnen die meisten Angriffe auf das Verzeichnis als Angriffe auf einzelnen Hosts. Vollständige Richtlinien zum Schützen von lokalen Gruppen auf Systemen, die Mitglied können nicht bereitgestellt werden, aber die folgenden Empfehlungen können verwendet werden, um die lokalen Administratorkonten auf Arbeitsstationen und Mitgliedsservern zu sichern.  

#### <a name="securing-local-administrator-accounts"></a>Schützen lokaler Administratorkonten

In allen Versionen von Windows, die derzeit in der mainstream-Support ist das lokale Administratorkonto standardmäßig deaktiviert, dadurch wird das Konto nicht vom Pass-the-Hash und anderen Angriffen mit gestohlenen Anmeldeinformationen. Allerdings können in Domänen, die ältere Betriebssysteme oder welche lokalen Administratorkonten aktiviert wurden, diese Konten verwendet werden wie zuvor beschrieben, um die Gefährdung über MemberServer und Arbeitsstationen weitergegeben werden. Aus diesem Grund werden die folgenden Steuerelemente für alle lokalen Administratorkonten auf der Domäne verbundenen Systemen empfohlen.  

Ausführliche Anweisungen zum Implementieren von diesen Steuerelementen finden Sie unter [Anhang H: Schützen lokaler Administratorkonten und-Gruppen](../../../ad-ds/plan/security-best-practices/Appendix-H--Securing-Local-Administrator-Accounts-and-Groups.md). Vor der Implementierung dieser Einstellungen, jedoch sicherstellen Sie, dass lokale Administratorkonten nicht gerade verwendet werden in der Umgebung zum Ausführen von Diensten auf Computern oder führen andere Aktivitäten, die für die diese Konten nicht verwendet werden soll. Testen Sie diese Einstellungen gründlich, bevor Sie Sie in einer produktionsumgebung implementieren.  

#### <a name="controls-for-local-administrator-accounts"></a>Steuerelemente für die Konten für lokale Administratoren

Integrierter Administratorkonten nie als Dienstkonten auf Mitgliedsservern verwendet werden soll, noch sollte sie zur Anmeldung beim lokalen Computer verwendet werden (außer im abgesicherten Modus, was zulässig ist, auch wenn das Konto deaktiviert ist). Das Ziel der Implementierung der hier beschriebenen Einstellungen wird verhindert, dass lokale Administratorkonto in jedes Computers wird verwendet, es sei denn, Sie Kontrollmechanismen zunächst rückgängig gemacht werden. Diese Steuerelemente implementieren und Überwachen von Administratorkonten für Änderungen, können Sie erheblich der Wahrscheinlichkeit des Erfolgs eines Angriffs, lokalen Administratorkonten der Ziele reduzieren.  

##### <a name="configuring-gpos-to-restrict-administrator-accounts-on-domain-joined-systems"></a>Konfigurieren von Gruppenrichtlinienobjekten über Administratorkonten für Domäne Angehörige Systeme einschränken

Fügen Sie das Administratorkonto in ein oder mehrere Gruppenrichtlinienobjekte, die Sie erstellen und auf Arbeitsstationen und Mitgliedsservern Server Organisationseinheiten in jeder Domäne verknüpfen, um in der folgenden Benutzerrechte **Computer Computerkonfiguration\Richtlinien\Windows unter Sicherheitseinstellungen\Lokale Policies\ Zuweisen von Benutzerrechten**:  

- Zugriff vom Netzwerk auf diesen Computer verweigern
- Anmelden als Batchauftrag verweigern
- Anmelden als Dienst verweigern
- Anmelden über Remotedesktopdienste verweigern

Wenn Sie diese Benutzerrechte für Administratorkonten hinzufügen, geben Sie, ob Sie hinzufügen, werden das lokale Administratorkonto oder den Domänenadministrator Konto übrigens, dass Sie das Konto bezeichnen. Z. B. hinzufügen, Verweigern der Domäne NWTRADERS Administratorkonto anmelden, um diese Rechte, geben Sie das Konto als **NWTRADERS\Administrator**, oder navigieren Sie zu dem Administratorkonto für der Domäne NWTRADERS. Um sicherzustellen, dass Sie das lokale Administratorkonto beschränken, geben Sie **Administrator** in diesen Zuweisen von Einstellungen in den Gruppenrichtlinienobjekt-Editor.  

> [!NOTE]  
> Auch wenn lokale Administratorkonten umbenannt werden, werden die Richtlinien angewendet.  

Diese Einstellungen werden sichergestellt, Administratorkonto des Computers kann nicht verwendet werden, auf die anderen Computer eine Verbindung herstellen, auch wenn sie versehentlich oder böswillig aktiviert ist. Lokale Anmeldungen, die mit dem lokalen Administratorkonto nicht vollständig deaktiviert werden, noch sollten Sie versuchen zu diesem Zweck einen lokalen Administrator Computerkonto ist darauf ausgelegt, Wiederherstellung im Notfall verwendet werden.  

Sollte ein Mitgliedsserver oder Arbeitsstation werden von der Domäne getrennt mit anderen lokalen Konten gewährt administrative Berechtigungen, der Computer im abgesicherten Modus gestartet werden kann, kann das Administratorkonto aktiviert werden und das Konto kann dann verwendet werden, um die Auswirkungen Reparaturen auf dem Computer. Beim Abschluss der Reparaturen sollte wieder das Administratorkonto deaktiviert werden.

#### <a name="securing-built-in-administrator-accounts-in-active-directory"></a>Schützen integrierter Administratorkonten in Active Directory

In jeder Domäne in Active Directory ist ein Administratorkonto im Rahmen der Erstellung der Domäne erstellt. Dieses Konto ist standardmäßig ein Mitglied der Domänen-Admins und der Administrator die Gruppen in der Domäne, und die Domäne der Gesamtstruktur-Stammdomäne, das Konto ist auch Mitglied der Gruppe "Organisations-Admins". Verwenden des lokalen Administratorkontos in einer Domäne sollten nur für die ersten Build-Aktivitäten und möglicherweise Notfallwiederherstellungsszenarien reserviert werden. Um sicherzustellen, dass ein integriertes Administratorkonto verwendet werden kann, um Reparaturen Auswirkungen auf den Fall, dass keine anderen Konten verwendet werden können, sollten Sie die Standardmitgliedschaft für das Administratorkonto in einer beliebigen Domäne in der Gesamtstruktur nicht ändern. Stattdessen sollten Sie folgende Richtlinien, um das Administratorkonto in jeder Domäne in der Gesamtstruktur zu schützen. Ausführliche Anweisungen zum Implementieren von diesen Steuerelementen finden Sie unter [Anhang D: Schützen integrierter Administratorkonten in Active Directory](../../../ad-ds/plan/security-best-practices/Appendix-D--Securing-Built-In-Administrator-Accounts-in-Active-Directory.md).  

#### <a name="controls-for-built-in-administrator-accounts"></a>Steuerelemente für integrierter Administratorkonten

Das Ziel der Implementierung der hier beschriebenen Einstellungen werden Administratorkonto für jede Domäne (keine Gruppe) zu verhindern, dass aus verwendet werden, es sei denn, eine Reihe von Steuerelementen vertauscht sind. Diese Steuerelemente implementieren, und die Administratorkonten auf Änderungen überwachen, können Sie die Wahrscheinlichkeit eines erfolgreichen Angriffs erheblich reduzieren, durch die Nutzung einer Domänen Administratorkonto. Für das Administratorkonto in jeder Domäne in der Gesamtstruktur sollten Sie die folgenden Einstellungen konfigurieren.  

##### <a name="enable-the-account-is-sensitive-and-cannot-be-delegated-flag-on-the-account"></a>Aktivieren Sie das Flag "Konto ist vertraulich und kann nicht delegiert werden" für das Konto

Standardmäßig können alle Konten in Active Directory delegiert werden. Delegierung ermöglicht, einen Computer oder ein Dienst, um die Anmeldeinformationen für ein Konto zu präsentieren, die auf dem Computer authentifiziert wurde oder Dienst mit anderen Computern aus, um Dienste für das Konto zu erhalten. Wenn Sie aktivieren die **Konto ist vertraulich und kann nicht delegiert werden** Attribut für ein domänenbasiertes Konto-Anmeldeinformationen des Kontos können nicht angezeigt werden sollen andere Computer oder Dienste im Netzwerk, die Angriffe einschränkt, die nutzen Delegierung der Anmeldeinformationen für die auf anderen Systemen zu verwenden.  

##### <a name="enable-the-smart-card-is-required-for-interactive-logon-flag-on-the-account"></a>Aktivieren Sie das Flag "Smartcard ist für die interaktive Anmeldung erforderlich" für das Konto

Beim Aktivieren der **Smartcard ist erforderlich, für die interaktive Anmeldung** Attribut für ein Windows-Konto setzt das Kennwort des Benutzerkontos auf einen zufälligen Wert von 120 Zeichen. Wenn Sie dieses Flag für integrierten Administrator-Konten festlegen, stellen Sie sicher, dass das Kennwort für das Konto nicht nur lange und komplexe ist, aber nicht für alle Benutzer bekannt ist. Kann nicht aus technischer Sicht erforderlich, Smartcards für die Konten zu erstellen, bevor Sie dieses Attribut zu aktivieren, aber wenn möglich, Smartcards, die für jedes Konto Administrator vor der Konfiguration der Konto-Einschränkungen erstellt werden soll und die Smartcards gespeichert werden sollen Sichere Speicherorte.  

Obwohl das Festlegen der **Smartcard ist erforderlich, für die interaktive Anmeldung** Kennzeichen setzt das Kennwort des Kontos, das es verhindert nicht, dass einen Benutzer mit Berechtigungen zum Zurücksetzen des Kennworts des Kontos, über das Konto auf einen bekannten Wert festlegen und Verwenden der Name des Kontos und neues Kennwort zum Zugriff auf Ressourcen im Netzwerk. Aus diesem Grund sollten Sie die folgenden zusätzlichen Steuerelemente für das Konto implementieren.  

##### <a name="configuring-gpos-to-restrict-domains-administrator-accounts-on-domain-joined-systems"></a>Konfigurieren von Gruppenrichtlinienobjekten zum Einschränken von Domänen-Administratorkonten auf Systemen mit Domänenverknüpfung

Obwohl das Konto deaktivieren des Administratorkontos in einer Domäne effektiv unbrauchbar ist, sollten Sie zusätzliche Einschränkungen für das Konto implementieren, für den Fall, dass das Konto, versehentlich oder böswillig aktiviert ist. Obwohl diese Steuerelemente letztendlich durch das Administratorkonto umgekehrt werden können, das Ziel ist, um Steuerelemente zu erstellen, die die verlangsamen eines Angreifers ausgeführt und Grenzwert der Schaden das Konto anrichten kann.

### <a name="role-based-access-controls-rbac-for-active-directory"></a>Rollenbasierte Zugriffssteuerung (RBAC) für Active Directory

Rollenbasierte Zugriffssteuerung (RBAC) sind im Allgemeinen einen Mechanismus zum Gruppieren von Benutzern und bietet Zugriff auf Ressourcen auf Grundlage der Geschäftsregeln. Im Fall von Active Directory ist die Implementierung von RBAC für AD DS den Prozess der Erstellung von Rollen, die auf die Rechte und Berechtigungen delegiert werden sollen, um die Mitglieder der Rolle alltägliche Verwaltungsaufgaben ausführen, ohne dass ihnen übermäßige Rechte zu ermöglichen. RBAC für die Active Directory kann entworfen und über systemeigene Tools und Schnittstellen, durch die Nutzung von Software, die Sie bereits besitzen können durch den Kauf von Drittanbieterprodukten oder eine beliebige Kombination der folgenden Ansätze, implementiert. Dieser Abschnitt bietet keine schrittweise Anweisungen zum Implementieren von RBAC für Active Directory, aber stattdessen erläutert Faktoren, die Sie berücksichtigen sollten, bei der Auswahl eines Ansatzes zum Implementieren von RBAC in Ihrer AD DS-Installationen.  

#### <a name="native-approaches-to-rbac-for-active-directory"></a>Native Ansätze zur RBAC für Active Directory

Bei der einfachste Implementierung von RBAC können Sie Rollen als AD DS-Gruppen zu implementieren und Delegieren Rechte und Berechtigungen für die Gruppen, die sie für die tägliche Verwaltung innerhalb des festgelegten Bereichs der Rolle zu ermöglichen.  

In einigen Fällen können vorhandenen Sicherheitsgruppen in Active Directory verwendet werden, Rechte und Berechtigungen für eine Funktion zu gewähren. Z. B. wenn bestimmte Mitarbeiter in Ihrer IT-Organisation für die Verwaltung und Wartung von DNS-Zonen und-Einträge verantwortlich sind, kann diese Aufgaben zu delegieren so einfach wie das Erstellen eines Kontos für die einzelnen DNS-Administratoren und dem DNS hinzufügt werden Gruppe "Administratoren" in Active Directory. Die DNS-Administratoren-Gruppe, im Gegensatz zu mehr sehr privilegierten Gruppen, verfügt über einige leistungsstarke Rechte in Active Directory, obwohl die Mitglieder dieser Gruppe zugewiesenen Berechtigungen wurden, die ihnen das Verwalten von DNS gewähren und unterliegen Gefährdung und Missbrauch verursachen rechteerweiterungen zu schützen.

In anderen Fällen müssen Sie zum Erstellen von Sicherheitsgruppen und Delegieren Rechte und Berechtigungen für Active Directory-Objekte, Dateisystemobjekte und Registrierungsobjekte können Mitglieder der Gruppen, um festgelegte administrative Aufgaben auszuführen. Z. B. wenn die Helpdesk-Operatoren für das Zurücksetzen vergessener Kennwörter verantwortlich sind, müssen Unterstützung von Benutzern bei der verbindungsherstellung und zur Problembehandlung in Anwendungseinstellungen, Sie zum Kombinieren von delegierungseinstellungen für Benutzerobjekte in Active Directory mit Berechtigungen, mit denen beim Helpdesk-Benutzer zum Herstellen einer Remoteverbindung mit Computern der Benutzer zum Anzeigen oder Ändern der Konfigurationseinstellungen des Benutzers. Für jede Rolle, die Sie definieren, sollten Sie Folgendes identifizieren:  

1. Welche Aufgaben Mitglieder der Rolle führen Sie auf die täglichen und welche Aufgaben weniger häufig ausgeführt werden.  
2. Auf welchen Systemen und in welche Anwendungen sollten Mitglieder einer Rolle Rechte und Berechtigungen gewährt werden.  
3. Welche Benutzer Zugehörigkeit zu einer Rolle gewährt werden soll.  
4. Wie wird die Verwaltung von Rollenmitgliedschaften ausgeführt werden.  

In vielen Umgebungen kann manuell die rollenbasierte Zugriffssteuerung für die Verwaltung von Active Directory-Umgebung erstellen schwierig sein, implementieren und zu verwalten. Wenn Sie Rollen und Zuständigkeiten für die Verwaltung der IT-Infrastruktur klar definiert haben, möchten möglicherweise nutzen Sie zusätzliche Tools, um Sie beim Erstellen einer verwaltbaren native RBAC-Bereitstellung unterstützen. Z. B. wenn Forefront Identity Manager (FIM) in Ihrer Umgebung verwendet wird, können FIM Sie zum Automatisieren das Erstellen und Auffüllen der Administratorrollen, die laufenden Verwaltungsaufgaben vereinfachen können. Bei Verwendung von System Center Configuration Manager (SCCM) und System Center Operations Manager (SCOM) können Sie anwendungsspezifischen Rollen Delegieren der Verwaltung und Überwachung von Funktionen verwenden, und auch die konsistente Konfiguration und Überwachung über Systeme in erzwingen die Domäne. Wenn Sie eine public Key-Infrastruktur (PKI) implementiert haben, können Sie ausgeben und benötigen Smartcards für IT-Mitarbeiter, die für die Verwaltung der umgebungs zuständig. Mit dem FIM-Anmeldeinformationen-Management (FIM CM) können Sie sogar Verwaltung von Rollen und Anmeldeinformationen für Ihre Administrationsmitarbeiter kombinieren.  

In anderen Fällen ist es möglicherweise besser für eine Organisation, erwägen die Bereitstellung von RBAC-Software von Drittanbietern, die "Out-of-Box" Funktionen bereitstellt.
